# Improvement Roadmap — 2026-07-21 ultracode audit

Produced by an 11-agent audit (5 subsystem readers, 5 improvement lenses, 1 synthesis judge). 80 frictions logged, 40 candidate ideas, verified against the code.

## Ranked roadmap

### 1. Meta results ingestion: CSV keyed to ad filename
**Impact:** transformative · **Effort:** day · **Lens:** ad-performance

**What:** Per-client 'Import Meta results' surface accepting a pasted/uploaded Ads Manager export (ad name, spend, CTR, CPL). Fuzzy-match on the stable export_drive filenames, stamp spend/CTR/CPL columns onto client_assets, stamp exportedAt at export time, and inject a 'MARKET TRUTH' tier above operator verdicts into the ads claim payload (server/workerRoutes.ts:351-373, scripts/workerLib.ts:80-103). Surface win chips + a 'Clone winner' button (pre-seeded Vary) in AssetGallery as phase 2.

**Why:** The single missing link for the core obsession. Today the calibration loop is 100% operator taste; one CSV paste per client per week upgrades every future batch from 'what I liked' to 'what actually converted'. No Meta API/OAuth — 90% of the value at 5% of the effort, and all the claim-injection plumbing already exists (verified).

**Builds on:** client_assets table, assetReviews injection in server/workerRoutes.ts:351-373 + scripts/workerLib.ts:80-103, stable filenames from runExport in scripts/worker.ts:397

### 2. Crash-safe, idempotent completion (spool + retry + two-phase write)
**Impact:** transformative · **Effort:** day · **Lens:** reliability-quality

**What:** Worker: spool the completed payload to disk before calling complete, retry-with-backoff, replay unsent spools on boot, delete jobDir only after a confirmed 200, and stop posting /fail when complete itself threw. Server: validate ALL contract docs BEFORE writing any (verified: validation currently happens inside the write loop at workerRoutes.ts:461-490, so a short doc N leaves docs 1..N-1 overwritten and PNGs discarded), short-circuit 200 if the job is already review/approved, and block /fail from overwriting review status. Write handler tests as part of this refactor.

**Why:** A network blip, Render cold start, or a body near the 50mb cap currently converts a fully-rendered multi-hour 4-Opus-session batch into 'failed' with temp dirs already deleted — the single most expensive failure in the system. This makes finished work unlosable to transport errors.

**Builds on:** scripts/worker.ts:385,417-429 (complete call + tick catch), server/workerRoutes.ts:401-542 (/api/worker/complete and /fail)

### 3. Self-healing job queue: atomic claim, heartbeat reaper, Mac online chip
**Impact:** high · **Effort:** day · **Lens:** reliability-quality

**What:** Rewrite claimNextQueuedJob to claim a specific row (UPDATE with a claim token, SELECT that exact row — verified the current code UPDATEs blind then reads back 'newest running', db.ts:734-747). Add jobs.heartbeatAt stamped by the existing 8s /api/worker/progress pings, a sweep that requeues 'running' jobs silent for 10+ min, lastClaimAt on every claim poll driving a 'Mac engine: online/offline since X' chip in AppShell, plus small cancel-queued and force-requeue mutations.

**Why:** Today a sleeping Mac leaves a job 'running' forever, generateStage returns CONFLICT, and the only escape is DB surgery — 'my Mac is asleep' looks identical to 'the app is broken'. Every autopilot idea (rank 5, auto-rebuild quickwin) depends on the queue being trustworthy first.

**Builds on:** server/db.ts:734 claimNextQueuedJob, /api/worker/progress + /api/worker/claim in server/workerRoutes.ts, jobs table

### 4. Keyboard-first ad review inside the lightbox
**Impact:** high · **Effort:** day · **Lens:** operator-leverage

**What:** Rebuild the AssetGallery lightbox into a review surface: arrow-key prev/next, A=approve, R=reject note (Enter submits), V=Vary, Esc=close, a '4 of 15, 9 pending' counter, auto-advance after each verdict, per-card feedback state (killing the shared textarea note-leak), and an 'Approve all remaining' with confirm. Verified purely frontend — reviewAsset and queueVariations mutations are already wired at engines.tsx:249-267.

**Why:** Ad review is the hottest loop in the operator's week: a 15-static batch costs ~45+ zoom-close-approve clicks today and recurs with every Vary/rebuild cycle. This collapses it to one sub-minute keyboard pass per batch, forever.

**Builds on:** AssetGallery + lightbox in client/src/components/engines.tsx:387-528; existing reviewAsset/queueVariations tRPC mutations

### 5. Auto-advance pipeline: approving a stage queues the next
**Impact:** transformative · **Effort:** day · **Lens:** automation

**What:** Per-client autoRun flag; when reviewStage approves and the flag is on, immediately createJob for the next stage in STAGE_ORDER (the gate passes by construction). Plus a 'Run pipeline' button that queues foundation the moment onboarding docs + research land. Ship AFTER ranks 2-3 so auto-queued work lands on a queue that can't strand or lose it.

**Why:** Onboarding-to-ads currently requires the operator to notice each approval and click Generate 4 separate times across days, with the Mac idle in between. Auto-advance means the pipeline runs overnight and mornings start with a review queue instead of a to-do list — the purest signed-to-shipped accelerant available.

**Builds on:** reviewStage + generateStage in server/routers.ts:602-696, STAGE_ORDER in server/stages.ts, ClientDetail.tsx stage cards; sequenced after ranks 2-3

### 6. Parse per-ad spec into structured columns at batch completion
**Impact:** high · **Effort:** day · **Lens:** ad-performance

**What:** At /api/worker/complete, parse the batch doc's per-filename spec lines (format, Reference:, sub-avatar, angle, awareness, hook category — already mandated as prose by server/stages.ts:128) into new columns on client_assets, with graceful fallback when parsing misses.

**Why:** Without this, rank 1's data dead-ends at 'ad_07 won' with no why. With it, 'UGC-tweet + problem-aware + curiosity hook wins in this niche' becomes a query. It is the schema foundation the QA scorer, future niche playbook, and winner analysis all sit on — one parser plus an ALTER.

**Builds on:** /api/worker/complete handler (server/workerRoutes.ts:401-542), ads batch-doc contract (server/stages.ts:128), Meta columns from rank 1

### 7. Independent QA + winner-score pass before operator review
**Impact:** transformative · **Effort:** days · **Lens:** ad-performance

**What:** After shards merge, run one cheap fast-model session (not the builder session) that views each PNG against its declared Reference:, the static-render-rules numbered gates, and a digest of accumulated approve/reject verdicts — emitting per-ad pass/fail + 0-100 score + one-line reason, stored on client_assets. AssetGallery sorts by score with fail badges; 'auto-rebuild fails' feeds the existing rejected-rebuild path. Backtest scores against real CTR once rank 1 data accumulates.

**Why:** Today visual QA is self-graded by the session that built the ad, so broken layouts reach the operator, and the review order is random. An independent grader catches the '12 good, 3 broken' class before review, puts likely winners first (fewer clicks), and the score IS a pre-spend prediction — the literal 'know which ads will win' ask, made falsifiable by the Meta backtest.

**Builds on:** scripts/worker.ts runJob post-merge hook, worker/frameworks/static-render-rules.md, calibration inputs in workerRoutes.ts:348-373, structured columns from rank 6, AssetGallery

### 8. Per-shard salvage + 'delivered 12 of 15' batch accounting
**Impact:** high · **Effort:** days · **Lens:** reliability-quality

**What:** Replace runJob's bare Promise.all (verified at scripts/worker.ts:361-379) with allSettled: merge succeeded shards, post them with a failure manifest, kill sibling claude processes on fatal failure. Server-side, count incoming PNGs and SPLIT units against the contract's expected N, stamp deliveredCount/expectedCount on the job, and offer one-click 'rebuild the missing 3' via the existing REBUILD flow.

**Why:** One shard timing out currently discards 10 completed ads' worth of Opus output and forces a full 90-minute re-run; a silently-dead shard delivers 10 of 15 with a clean 'review' status, discovered by counting thumbnails. Turns a lost batch into a lost shard and makes partial delivery visible and cheaply repairable.

**Builds on:** scripts/worker.ts runJob/planShards, server/workerRoutes.ts complete handler, contract counts in server/stages.ts, existing REBUILD-rejected flow

### 9. Typed job payloads + single-sourced shard/feedback templates
**Impact:** high · **Effort:** day · **Lens:** reliability-quality

**What:** Move the 'Generate EXACTLY N NEW static ads' and 'REBUILD ONLY' templates (verified composed in engines.tsx:317/579 and regex-parsed in worker.ts:205-275) into shared constants imported by both sides; promote mode ('new'|'rebuild'|'variations'), count, and rejected filenames into typed jobs.payload fields the server composes into prose. Unit-test planShards (a pure function) for all three shard cases + fallthrough.

**Why:** Any phrasing drift silently disables sharding and collapses a 4-way parallel render into one 90-minute mega-session risking the SIGKILL timeout — the brittlest contract in the repo, flagged independently by three audit lenses. Typed payloads also let the server validate that a rebuild can't accidentally become a full 15-ad batch, and make the auto-rebuild quickwin safe.

**Builds on:** client/src/components/engines.tsx feedback composers, scripts/worker.ts planShards, server/stages.ts, jobs.payload JSON column

### 10. Tokenized read-only client portal at /portal/:token
**Impact:** transformative · **Effort:** days · **Lens:** client-experience

**What:** Clone the existing report share-token pattern (verified: share router at routers.ts:1015 + getSharedReportByToken + public /share/:token route) into a client-scoped portal: client_portal_tokens table, a public portal.get returning ONLY approved/posted docs and approved assets for one clientId, and a clean branded Portal.tsx (foundation docs, email sequences, ad gallery). One revocable magic link per client; deliberately a separate query path so nothing operator-only leaks. No auth build.

**Why:** Today clients see work only via Drive dumps and pasted DMs. A permanent 'your ads live here' link makes the agency feel like a product, kills 'can you send me the ads?' messages, and is the foundation the weekly report, recording queue, and client-feedback ideas all stack on — while deferring the whole logins/invites build.

**Builds on:** share router + SharedReport.tsx token pattern (server/routers.ts:1015, client/src/App.tsx:43), client_documents kanban status, approved-assets filter in server/db.ts

### 11. Weekly auto-report: what shipped, what's live, how socials moved
**Impact:** high · **Effort:** day · **Lens:** client-experience

**What:** A Monday cron handler next to trendRefreshHandler (same CRON_SECRET auth) composing a per-client digest from data already in the DB — docs approved/posted that week, ads approved/exported, engines run, latest social-stats snapshot — stored as a client_document so it appears in the portal and Overview automatically. Template-rendered v1 (zero LLM cost); optional worker-polished prose later.

**Why:** Retention for a DFY agency is 'do they feel the machine working'. A Monday report the operator never writes — '15 ads, 14 emails, 3 reels shipped' — is the cheapest churn insurance available, reusing the cron pattern, doc store, and portal with no new infrastructure.

**Builds on:** trendRefreshHandler pattern in server/scheduledHandlers.ts (notifyOwner verified imported), client_documents store, jobs/client_assets timestamps, portal from rank 10, social-snapshot quickwin

### 12. Auto re-mine research + feed trend snapshots into every batch
**Impact:** high · **Effort:** day · **Lens:** automation

**What:** Extend the auto-intel handler (workerRoutes.ts:638, already iterating all clients on the worker's 6h ping): if a client's linked analysis is older than N weeks, requeue scrape + runAnalysis (24h cache and dedup already exist). And join the latest trendSnapshots row into renderResearchForClient as a '## TRENDING THIS WEEK' section — the weekly cron already produces this data; it never reaches a prompt.

**Why:** A month-3 ad batch is currently written from month-0 customer language, and the weekly trend cron burns SerpAPI/YouTube quota producing hook fuel only a UI tab ever sees. Pure plumbing on existing data that upgrades every future batch's hooks with zero operator effort.

**Builds on:** auto-intel handler (server/workerRoutes.ts:638), renderResearchForClient (workerRoutes.ts:120), trendSnapshots from server/scheduledHandlers.ts, scrape cache in server/db.ts:477

## Considered and rejected (with reasons)

- Batches as designed test matrices — premature: without Meta outcome data (rank 1) and structured columns (rank 6), a matrix can't be read as 'hook B beat hook A'; the useful half (typed payloads) is rank 9
- Cross-client niche playbook — needs structured columns plus months of verdict/performance volume across 2+ same-niche clients; revisit once ranks 1 and 6 have accumulated data (do cap the unbounded assetReviews payload cheaply in the meantime)
- Named-competitor Meta Ad Library spy — real signal but days of new scraping/API surface; the Foreplay relevance-gate quickwin covers the freshest need now, revisit after the performance loop closes
- Scheduled fresh-ads drip — automates spending Max-plan Opus capacity before the queue self-heals (rank 3), completion is crash-safe (rank 2), and QA gates exist (rank 7); dangerous to ship first
- Cross-client review inbox / Dashboard rebuild — the morning-digest quickwin delivers ~70% of 'what needs me' in hours vs days of Dashboard + URL-state work; revisit when client count makes a push digest insufficient
- Self-serve onboarding intake link — onboarding happens once per client; the docx-accept + client-edit quickwins remove most of the pain at a tenth of the cost
- Client approve/request-changes in portal — wait until the portal (rank 10) proves clients actually open the link before building interactive verdict flows on it
- Client proof-and-testimonial uploads — portal-dependent; fold in as the first portal extension (it's hours once rank 10 exists)
- Recording queue for scripts — genuinely high-value but strictly sequenced: needs the portal shipped AND adopted; make it the second portal extension, not a standalone build now
- Content calendar revival — days of wiring for a surface neither operator nor clients view today; revisit post-portal when dates have an audience
- Job timeline / job_events table — observability polish; the heartbeat reaper (rank 3) and delivered-N-of-M accounting (rank 8) cover the operational need, and a slim jobs-status poll can ride along with rank 3
- One-click GHL/ConvertKit sequence export — real but lower-frequency than the ad loop; the token-guard quickwin removes the dangerous half, full formatted export is a later nice-to-have
- Ad run-history UI + Clone winner as a standalone item — it's phase 2 of rank 1 (Meta ingestion), folded in there
- Standalone regression-test suite for /api/worker/complete — write the handler tests as part of the rank-2 and rank-8 refactors that touch that code anyway; the deterministic-suite and healthcheck bits are quickwins
- Batch-scoped Drive export with folder links — the exportedAt stamp rides along with rank 1; the folder-link and per-batch selection polish can wait
- Full worker-queue drawer UI — merged into rank 3 (online chip + cancel/requeue mutations); a polished drawer is cosmetic once the queue self-heals
