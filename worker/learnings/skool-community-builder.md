# Learnings: skool-community-builder

Accumulated craft lessons applied on every run.

<!-- 2026-07-09 · job 60001 · client: Trent Kus -->
- The skill's own name and header brand it "Full Stack Agency," but the mother skill (client-onboarding-orchestrator) mandates that the agency is "Cashflow Coaches" and that "Full Stack Agency" must never appear in any deliverable. Update the skill's branding to Cashflow Coaches (or add a note that the internal name is legacy and never ships) so a writer following this skill doesn't leak the wrong agency name.
- FULL BUILD produces one community per run, but many client funnels need a matched pair: a free top-of-funnel community and a paid delivery community for the offer. The skill treats free-vs-paid only as a VSL variable. Add an explicit "dual build" note covering how to name the pair so ascension reads cleanly, how to differentiate their About pages (free sells one action, paid sells the offer/price/guarantee), and how to link the free community's ascension line to the paid group.
- The membership-question segmenter options are written for a pre-funding credit lead. For a PAID membership the segmenter should split by in-program stage (e.g. repair stage, close-to-fundable) rather than journey-to-entry. Add one line telling the writer to re-cut the segmenter options to the community's tier and the member's actual state on entry.
- Section 4 hands the VSL to `vsl-and-sales-page-writer`, but that skill is not always in scope for the Skool stage (it is a Step 1/Step 3 child in the orchestrator). State that when the handoff target is unavailable, this skill writes the VSL directly from the archetype brief here and still runs the humanizer/dimensionalization chain, so the step never stalls.

<!-- 2026-07-09 · job 90001 · client: Trent Kus -->
- The Section 3 About-page template hardcodes a "⚡ Welcome to [Community Name]" opener, but the appendix instructs the writer to pick the opening move that fits the client (identity / problem-agitation / guarantee / proof / mechanism / mission). These conflict. State that the fixed template skeleton is a fallback only, and the appendix's opening-move selection overrides the default "Welcome" line, so the strongest hook (a hard proof line for a paid page, problem-agitation for a credit free page) leads instead of a generic greeting.
- In a paired free-plus-paid build, the "Start Here" pinned post's first action differs by tier and should not be copied verbatim across both. Free = "Start Here & Book Call"; a paid delivery community's first action is usually an onboarding/setup step (here, "Set Up Monitoring"), not a call. Add a note that the first pinned post's action is retargeted to the tier's actual first step, while Steps 2 and 3 (Introduce Yourself, Accountability Buddy) stay fixed.

<!-- 2026-07-09 · job 90002 · client: Trent Kus -->
- Section 6 (gamification) says Level 1 is Caution and Levels 2 to 9 "progressively reflect growth," but it never states the cross-community rule that a paired free-plus-paid build requires. Add it: in any matched free-plus-paid build, the 1 to 9 level ladder must be IDENTICAL across both communities (one identity system, derived from the client's brand language), so a member ascending from free to paid keeps climbing the same ladder instead of resetting. Without this the skill produces two different ladders and the ascension reads as two disconnected journeys.
- Section 4 (About-page VSL brief) sets tone and archetype but gives no length target, so free VSLs get written too short (under 2 minutes) and paid ones drift long. Add the target from the winning-community data: about 3 minutes spoken, roughly 420 to 480 words, for BOTH the free and paid About-page VSL, compressing beats rather than skipping them. A stated word-count band keeps the two pages congruent and stops the free VSL from underselling.

<!-- 2026-07-11 · job 270011 · client: Trent Kus -->
- The skill produces launch assets only (name, tagline, About page, VSL, categories, level names, plugins, and the three fixed pinned posts) but has no mode or templates for ongoing / on-demand community CONTENT posts. A batch of value, engagement, win/proof, and DM-trigger posts is a recurring deliverable and currently has to be modeled cold from the research report's proven-posts section. Add a "content posts" mode covering the four post types, the comment-keyword DM-workflow skeleton (keyword, first message, qualification exchange, booking push to the funnel link), the one-win-per-asset proof-gate discipline, and the rule that every post must reference the community's real pinned content and its 1-to-9 level ladder so the batch reads as one build with the community. This is true for every client, not this one's taste.
