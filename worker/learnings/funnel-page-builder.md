# Learnings: funnel-page-builder

Accumulated craft lessons applied on every run.

<!-- 2026-07-09 · job 60002 · client: Trent Kus -->
- There is no mode for the VSL landing page itself, yet a call funnel (Branch B of the orchestrator) always needs one: the page the VSL sits on with headline, video embed, single booking CTA, proof strip, and mechanism block. LEAD_MAGNET, POST_CALL_PAGE, PRE_CALL_HUB, and CALENDLY all assume the lead is already past the VSL. Add a VSL_PAGE mode (or an explicit note) so the writer has a defined structure for the top-of-funnel VSL page instead of improvising it.
- POST_CALL_PAGE and PRE_CALL_HUB both live after the booking and overlap heavily, but the skill never states how they relate in sequence. Clarify that the post-call page is the immediate confirmation page (headline confirming the booking, call-confirmed video, offer-breakdown video, a few breakouts, on-page wins) and the pre-call hub is the deeper linked page reached from it (wall of wins, profile-sorted interviews, full offer walkthrough), so a writer producing both knows the intensity ramp and does not duplicate sections.

- There is still no VSL_PAGE mode, yet a Branch B call funnel always needs one and the funnel-structure framework already specifies its 14-section anatomy. Add an explicit VSL_PAGE mode that points the writer to the framework's spec, so the top-of-funnel VSL landing page is built to structure instead of improvised. LEAD_MAGNET, POST_CALL_PAGE, PRE_CALL_HUB, and CALENDLY all assume the lead is already past the VSL.
- The CALENDLY mode defines the four booking questions but the funnel-structure framework's Page 2 (booking page) anatomy does not reference them, so a writer building the booking page from the framework alone can miss them. Cross-reference the two: the booking page's four questions ARE the CALENDLY set, and they belong on Page 2, not on a separate step.

<!-- 2026-07-09 · job 120004 · client: Trent Kus -->
- There is still no VSL_PAGE mode, yet a Branch B call funnel always needs one. The VSL landing page (headline, subhead, VSL embed, single booking CTA, proof strip, mechanism, offer stack, testimonials, qualifier, FAQ, cost-of-inaction close) is improvised every run. Add an explicit VSL_PAGE mode that points to the funnel-structure framework's 14-section anatomy, and note that POST_CALL_PAGE / PRE_CALL_HUB are the pages downstream of it.
- The framework's gold-standard swipe (The Empowered Man) places the founder story on Page 1 BEFORE the pitch, but the funnel-structure 14-section anatomy has no founder-story section, so a writer building strictly to the anatomy drops it to the booking page. Add an explicit founder-story slot to the VSL-page anatomy (after the not-your-fault bridge, before the offer stack) for burned-before / high-trust-burden markets, and note it should weld to the differentiating offer deliverable where one exists.

<!-- 2026-07-09 · job 120004 · client: Trent Kus -->
- CALENDLY mode defines the four booking questions but neither it nor the framework's Page 2 anatomy specifies routing logic for the low-intent answer on Q3 ("Just exploring for now"). Add a rule: route that answer to the community link, not the calendar, so it does double duty as a show-quality filter rather than letting tyre-kickers book.

<!-- 2026-07-10 · job 180001 · client: Trent Kus -->
- The booking-page guidance is incomplete for a Branch B call funnel. CALENDLY mode supplies the four questions but nothing on flow architecture, so a writer building strictly to it produces a naive two-page booking flow. Add to the booking-page structure: (1) the application and scheduler are ONE integrated step (time-slot selection is the final form field, never a separate "pick a time" page, because splitting them costs roughly 50% drop-off and doubles cost per call), and (2) a disqualified (DQ) outcome page distinct from the confirmation page, with a soft next step (community/drop-sell) and the pixel rule that Schedule / Complete Registration fires only on the qualified confirmation page. Cross-reference call-funnel-blueprint, which carries these rules today while funnel-page-builder does not.

<!-- 2026-07-12 · job 270017 · client: Trent Kus -->
- The still-missing VSL_PAGE mode should specify that the page deliverable includes a rendered, previewable HTML build appended after the section-by-section copy (the operator downloads it or imports it into GHL), not only the Gamma-style structure. LEAD_MAGNET/POST_CALL_PAGE/PRE_CALL_HUB all output structure-for-a-designer; the top-of-funnel VSL landing page additionally requires the finished responsive HTML. Add this so the writer builds the page for real instead of stopping at a spec.
- Any multi-column comparison grid on a responsive landing page must be authored as TWO renders in the HTML: a real N-wide grid for desktop and a separate stacked row-block version revealed by a mobile `@media` query. CSS-shrinking one 3-wide table onto a phone destroys the contrast that carries the section. Bake the two-render rule into the page-build guidance.
- `[PROOF: ...]` and photo slots in the built HTML must render as visibly empty bordered placeholders (dashed border, label text), never a fabricated image, so the screenshot gate stays a hard publish blocker inside the previewable page and the operator can see exactly what asset to drop where.

<!-- 2026-07-14 · job 360010 · client: Blake Matthews -->
- On a two-page book-a-call funnel, resolve the price-placement tension explicitly: the cold VSL landing page and the VSL script name NO dollar price (book-a-call funnels never price in cold traffic), but the post-booking Offer Breakdown video is a legitimate and expected place to present the full price STRUCTURE, because that viewer has already booked and is in justification mode. State this split in the skill so the builder does not either plaster price on the cold page or strip it from the warm offer-breakdown where the prompt asks for "pricing."
