# Learnings: static-ad-builder

Accumulated craft lessons applied on every run.

<!-- 2026-07-10 · job 180009 · client: Trent Kus -->
- The skill documents only the SVG + rsvg-convert + components.py pipeline, but the live agency build route is the ad-factory HTML pipeline: one HTML file of `.s` design elements at 360x450, rendered to 1080x1350 via puppeteer (`render.js`, deviceScaleFactor 3), plus `genimage.js` (Nano Banana Pro) for photo-real formats. Worth documenting as the supported alternative so a builder does not have to reverse-engineer it.
- Add a hard warning about CSS class collisions when many `.s` ads live in one file: an unscoped utility class reused across ads (e.g. a generic `.big` used both as a full-ad wrapper and as an inline number style) bleeds background, padding, and flex onto the wrong element and silently overflows the crop. Rule: scope every per-ad class under that ad's id or a unique prefix, and pixel-audit each render for right-edge clipping.
- For lifestyle / photo + caption-chip formats, recommend generating the photo with genimage (text-free) and compositing the caption chips as HTML over a `file://` background, then rendering. Baking caption text into the image model is unreliable; the HTML overlay guarantees text fidelity, which the format needs.

<!-- 2026-07-10 · operator rejection · client: Trent Kus · batch AdsBatch_2026-07-10 -->
- Operator graded the batch 5/10 and rejected it. The four failures, now codified as hard gates in frameworks/static-render-rules.md: (1) hand-rolled HTML chrome instead of this skill's own pipeline (scripts/ + assets/chrome real-device crops + fonts): for covered formats the skill's pipeline is mandatory, ad-factory is fallback only; (2) designed colored CTA text ("link below" in brand green/blue) baked onto native formats: CTAs live ONLY in the format's designated CTA slot (marker line, final chat bubble, last caption chip) styled per spec; (3) an AI-generated person stood in for the founder on the lifestyle ad: never generate humans, real client photos or a no-person composition only; (4) a tweet ad shipped with a dead white void because no one looked at the render: the visual QA loop (render, VIEW the PNG, grade against reference screenshots and the native test, fix, re-render) is mandatory per ad.

<!-- 2026-07-10 · job 210001 · client: Trent Kus -->
- The QA step (Step 4) checks horizontal extents (`qa_extents`, nothing past x=1012) but has no check for VERTICAL overflow. Fixed-height phone-native formats (iMessage, WhatsApp, IG DM) with `overflow:hidden` silently clip the last message or the footer with no error, dropping the closing beat. Add an explicit clip check to the QA loop: render, then confirm the last authored element is fully visible, or use flow layout that grows the canvas instead of a hard-coded height.
- The genimage guidance for lifestyle/story formats should state plainly: environment-only, no person, no figures, no silhouettes. The image model inserts a human by default even when the brief says "founder from behind," which collides with the "NEVER AI-generate a human" red line. Reserve any human founder shot for the client's real supplied photo, and spec the AI-generated frame as objects and environment only.

<!-- 2026-07-11 · job 210003 · client: Trent Kus -->
- Chrome compositing is silently reversible by any later render. `render_all()` regenerates every PNG in `WORKDIR` from its SVG, so if a second generator script (e.g. a non-Notes `gen_native.py`) calls `render_all()` after `gen_notes.py` has already run `compose_chrome()`, the real device chrome is stripped back off the Notes PNGs and they ship as bare white/black notes. This is easy to miss because the SVG never changed. Fix options, any one: (a) have `compose_chrome()` write to a distinct `*_final.png` name instead of overwriting the render, (b) make `render_all()` skip files recorded in a "chrome already composited" manifest, or (c) add a hard warning in SKILL.md Step 4/5 that `compose_chrome()` must be the LAST operation to touch every Notes PNG, after ALL `render_all()` calls across every generator in the batch. A one-line note in the SKILL would have prevented a re-composite pass this run.
- `notes_v2.py` resolves its fonts/chrome via `_HERE = os.path.dirname(os.path.abspath(__file__))`, which breaks when the module is imported from a copied/relocated build dir; the prior client run had to ship a `notes_v2_local.py` shim that hardcodes the absolute skill path. Consider resolving the assets path from the installed skill root (or an env var) so callers can import the real `notes_v2` from any working directory without a shim.
- The visual QA loop is the one rule that catches font-metric wrapping bugs: a hand-built ChatGPT-answer fallback format clipped its inline step lines off the right edge on first render, and only viewing the PNG surfaced it. Worth adding an explicit line to the QA section that any NON-skill fallback format (ad-factory / hand-built SVG) must be width-audited with `text_w()` before its first render, since those formats do not get the components' built-in extent helpers for free.

<!-- 2026-07-11 · job 240001 · client: Trent Kus -->
- A visible, high-contrast CTA is not optional on any native format that has a CTA slot (Reminders, Email, Notes, chat threads). A missing CTA, or a CTA rendered in low-contrast grey, is an operator-level rejection ("an ad without a proper CTA = burning money," "that greyed out text is not eye catching, what's the point"). Render the CTA in the format's own accent (iOS link blue, the marker line) so it reads as a real tappable action, not a caption.
- When a format imitates a recognizable app UI, match the REAL app, not an approximation. A calendar ad must use an authentic Apple or Google Calendar day view (weekday-and-date strip, hour grid, solid event blocks, current-time line), not two floating colored blocks. Half-native UI reads fake and gets rejected; if you cannot get the real UI right, switch formats.
- A proof-gated ad (a [PROOF] screenshot that drops in at launch) must look intentional in its empty state, never a giant dashed placeholder box that dominates the frame. Make the number or the story the hero and keep the screenshot slot a compact, clearly-labeled frame near the foot, so the pre-screenshot render still reads as a finished ad and not a wireframe.
- Do not let a direct-response designed format drift into a corporate or SaaS aesthetic (glossy white cards, pill headers, a pricing-table look). It stops matching the client's own creative language and reads as an obvious ad. For a comparison, default to the handmade whiteboard (marker, red crosses vs green checks, hand-drawn divider) or the native iOS Notes table before a designed card grid.
- Render order for any batch that mixes ad-factory/SVG natives with composited-chrome Notes ads: render the natives first, composite the Notes chrome last. `render_all()` re-rasterizes every SVG in the workdir, so a later native re-render silently strips the chrome that `compose_chrome()` already pasted onto the Notes PNGs.

<!-- 2026-07-11 · job 240001 · client: Trent Kus -->
- Render the on-image CTA in a high-contrast, eye-catching treatment (the blue link-style "Click the link below" the client uses), never greyed-out or low-contrast. A muted CTA that blends into the layout gets flagged as pointless by the operator even when the rest of the ad is approved. Contrast on the CTA is a render-QA gate, not a style preference.

<!-- 2026-07-11 · job 270001 · client: Trent Kus -->
- Search-format ads must match the query to the viewer's awareness level. A branded "is <company> legit / reviews / scam" search is a most-aware prospect vetting a company they already know, so it only works on retargeting. For cold, problem-aware traffic the query must be the pain the prospect actually types (e.g. "why do i keep getting denied for funding"), with the autocomplete rows as adjacent pain searches, and the result block as the solution reveal.
- A search result must carry the full native anatomy or the illusion breaks: favicon, site name, URL breadcrumb, a blue title, then exactly ONE grey description paragraph. Orphaned lines of text floating with no chrome read as a graphic, not a screenshot.
- On notes-app-family formats (Notes, Reminders, checklists), the CTA is handwritten below or outside the app frame, never a fake in-app list item. A checked-circle "Click the link" row breaks the UI because those apps never contain CTAs.
- On any phone-native recreation, a typed blue/underlined CTA makes the creative "look like an ad twice." The CTA belongs either in the caption/primary text or as a handwritten marker scrawl outside the UI, never as designed CTA text floating inside the screenshot.
- "Click the link below" is correct only on native formats where the caption link sits beneath the creative. On a designed DR graphic where the button IS the link, that phrasing reads as confused; use an outcome-worded button label instead.
- When rebuilding to a named handmade reference, match that reference's execution, not just its copy. `mm_var1_napkin_math` is off-white lined notebook paper with every word handwritten (red kicker, equation flow, yellow-underlined green payoff, arrow CTA), not a photoreal napkin with black letterbox bars. `v4_static_editorial` is defined by an editorial serif with an eyebrow (thin rule plus letterspaced caps) and a bordered pill CTA, not bold sans-serif. A comparison ad in a handmade brand system is a marker whiteboard, not a SaaS-style card table.
- Never ship placeholder text baked into a final rendered creative. An on-image line like "screenshot drops in here before launch" means unfinished work was submitted for review, and it leaves the ad with zero proof. Either the real asset is on screen or the concept ships in a format that stands on its own.
- Match reference UI chrome icon-for-icon on branded third-party formats (ChatGPT, iMessage, etc.). Wrong or warped action icons and stray icons not in the reference (e.g. an extra share/upload arrow) get the recreation clocked as fake. Pull the exact icon row and input-bar layout from the reference screenshot.
