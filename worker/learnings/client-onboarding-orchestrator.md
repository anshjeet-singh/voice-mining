# Learnings: client-onboarding-orchestrator

Accumulated craft lessons applied on every run.

<!-- 2026-07-07 · job 1 · client: Trent Kus -->
- Add an explicit content-safety pass to the voice rules: raw client onboarding material (voice memos, call transcripts) frequently contains profanity, offensive remarks, or admissions of non-compliant practices. Deliverables must extract the client's energy and phrasing while filtering that content out, defaulting to the client's professional/on-call register rather than their raw private register. The current rules say "write in the client's voice" but never say to sanitize it, which risks leaking unpublishable content into assets.

<!-- 2026-07-09 · job 30001 · client: Trent Kus -->
- The Step 1 format rule tells the writer to open the exemplar client's `Foundations/` folder in Google Drive and match its structure. In local, headless, or no-Drive-connector runs that folder is unreachable, which can stall the step or lead to guessing. State explicitly that the embedded Appendix templates are the authoritative source of truth for shape, and the Drive exemplar is a cross-check to use only when reachable, so a run never blocks on Drive access.
- The Appendix says "no horizontal-rule dividers" but this is a Google Doc filing rule, while the same docs are frequently delivered first as rendering markdown (web app preview) where `---` between major sections aids scannability. Clarify that the no-horizontal-rule constraint applies to the Google Doc upload, not the markdown deliverable, so writers aren't caught between two conflicting formatting instructions.

<!-- 2026-07-09 · job 150001 · client: Trent Kus -->
- The Step 5 SMS spec is written by the orchestrator directly with no child skill, but it never defines placeholders for the two links SMS actually needs: the meeting/join link and the pre-call hub link. The funnel-structure framework only standardizes `[BOOKING LINK]`, `[VSL LINK]`, and `[COMMUNITY LINK]`. Add `[CALL LINK]` (the calendar event's join link, used by the reminder texts) and `[PRE-CALL HUB LINK]` to the canonical placeholder set so SMS reminders and the asset nudge aren't improvising URLs.
- The SMS rules list voice and length constraints but omit the TCPA opt-out requirement. Add a note that the first message in any SMS set should carry a short opt-out ("Reply STOP to opt out"), since it does not count as the one CTA and is a legal requirement for marketing SMS.

<!-- 2026-07-13 · job 330001 · client: Blake Matthews -->
- There is a direct conflict between the global formatting rule "no em dashes anywhere" and the required `### Name — descriptor` sub-avatar heading, which the app parses on the dash. The heading dash is load-bearing for parsing and must be kept; the no-em-dash rule should be scoped explicitly to body prose and non-parsed headings, so writers do not strip the dash the parser depends on.
- The ICP Snapshot's 11-section fixed template lists "Sub-Avatars tables with %" but the schwartz-awareness-mapper output (awareness level + messaging rules per sub-avatar) has no named home in that list. In practice the awareness map is a required carried-forward artifact and belongs in the ICP doc. Name an explicit "Awareness Stage & Messaging Map" section (plus per-sub-avatar awareness level) in the ICP template so the artifact is produced consistently instead of being improvised each run.

<!-- 2026-07-13 · job 330001 · client: Blake Matthews -->
- The Step 1 fixed-template rule enumerates the section list for ICP Snapshot (11 sections), Brand & Positioning (14 sections), and Course Outline, but for Offers it only says "follows the format of the exemplar Offers doc exactly." In headless / no-Drive runs the exemplar is unreachable, so the writer has no embedded spec for the Offers doc and must reconstruct the shape from the child skills, whose native output formats do not match (offer-extraction outputs "Offer Angles / Dominant Offer," offer-architecture outputs an 8-part audit, low-ticket-to-high-ticket outputs a full ascension plan). Embed an authoritative Offers spec in the mother skill: an offer ladder of exactly three rungs (Free / Low-Mid / High), each as its own `## ` section, with name, price, promise, named mechanism, and guarantee, plus offer math on the high-ticket rung. This makes the Offers doc reproducible without the Drive exemplar, matching how ICP/Brand/Course are already specced inline.

<!-- 2026-07-13 · job 330001 · client: Blake Matthews -->
- The Brand & Positioning 14-section template names "Awareness Levels & Big Domino" but gives no instruction to tense the section to the funnel type. For a webinar funnel the Big Domino is the single belief the live room must accept before the pitch, and the awareness ladder should end at "book the call / take the spot"; for a call funnel it ends at the VSL-to-application handoff. Add one line telling the writer to frame the awareness map and domino to the funnel branch selected in Step 0.

<!-- 2026-07-13 · job 330002 · client: Blake Matthews -->
- The Offers doc must render as exactly three `## ` sections in the order Free, Low/Mid, High because the app maps those headings to the offer selector. When an operator hands over a preferred offer-doc layout (e.g. High Ticket first as "Part 1," a glance table, or renamed section headers), the fixed three-section order and heading names still win for the parser; the operator's ordering and detail get folded INTO the matching rung as `###` subsections, not allowed to override the `##` structure. State this precedence in the embedded Offers spec so a regeneration against operator-supplied formatting does not break the selector mapping.
- The embedded Offers spec lists only minimal per-rung fields (name, price, promise, mechanism, guarantee, plus offer math on high). In practice the high-ticket DFY rung needs the fuller Hormozi/Godfather stack to pass the quality bar: named two-half mechanism, an itemized value-stack table with per-item dollar values summed to a stack total, the price structure, a value-equation table (dream outcome / likelihood / time / effort), a named bonus stack whose value eclipses the core, the guarantee, and real scarcity. Add these as the expected `###` subsections of the High rung so the doc is not reconstructed as five thin bullets each run.
- For regulated niches (peptides, supplements, health, finance) the Offers doc itself should carry the compliance line (research use only, no human/medical claims, every income figure framed as a real documented result not a promise) as client-facing positioning, not as bracketed internal notes, since the "no internal working notes" document rule otherwise pushes writers to strip the guardrail the niche legally needs.
